\chapter{Reproduction du papier}

\section{Objectif de la reproduction}

L'objectif de cette partie du projet etait de reproduire aussi fidelement que
possible les resultats presentes dans l'article
\emph{Dial-In LLM: Human-Aligned LLM-in-the-loop Intent Clustering for Customer
Service Dialogues}~\cite{hong2025dialinllm}, en se concentrant sur la partie la
plus reproductible avec les ressources publiques : l'evaluation sur les
benchmarks anglais.

Plutot que de viser immediatement les resultats principaux sur le corpus
chinois des auteurs, nous avons choisi de cibler le tableau de comparaison sur
quatre benchmarks anglais, car :
\begin{itemize}[leftmargin=*]
  \item les jeux de donnees sont publics ;
  \item les tailles de benchmark sont connues ;
  \item les scores cibles du papier sont explicitement rapportes ;
  \item cette partie permet deja d'evaluer la qualite du pipeline
  LLM-in-the-loop dans un cadre comparable.
\end{itemize}

  protocole completement tracable.
\end{itemize}

\section{Environnement experimental}

Les experiences de reproduction ont ete executees sur le serveur Jupyter du
projet, dans le repertoire \texttt{/nfs/LLM-DIAL}.

Les contraintes retenues etaient les suivantes :
\begin{itemize}[leftmargin=*]
  \item utilisation systematique du second GPU \texttt{cuda:1} ;
  \item ecriture uniquement dans le depot du projet ;
  \item caches modeles conserves dans \texttt{.hf-cache/} ;
  \item sorties de runs conservees dans \texttt{out/} ;
  \item aucune modification de fichier ou de depot hors du projet.
\end{itemize}

L'environnement observe etait :
\begin{itemize}[leftmargin=*]
  \item Python 3.12.6 ;
  \item \texttt{torch} disponible ;
  \item \texttt{transformers} disponible ;
  \item \texttt{sentence-transformers} disponible ;
  \item deux GPU NVIDIA L40 ;
  \item \texttt{accelerate} non disponible ;
  \item \texttt{bitsandbytes} non disponible.
\end{itemize}

Ces contraintes ont directement influence le niveau de fidelite accessible par
rapport au papier.

\section{Choix d'implementation}

L'article indique que les benchmarks anglais sont traites avec un
\texttt{Llama-7B} fine-tune sur 800 clusters d'intents annotes. En pratique, la
reproduction exacte de cette configuration n'etait pas possible dans notre
environnement, car les checkpoints officiels equivalents Meta-Llama sont
proteges par un acces \emph{gated}, et les modeles fine-tunes des auteurs ne
sont pas publics.

Nous avons donc retenu la pile suivante comme approximation la plus fidele
reellement executable :
\begin{itemize}[leftmargin=*]
  \item embeddings : \texttt{BAAI/bge-large-en-v1.5} ;
  \item evaluateur et namer locaux :
  \texttt{mistralai/Mistral-7B-Instruct-v0.3} ;
  \item clustering : \texttt{MiniBatchKMeans} ;
  \item \texttt{sample-size = 10} ;
  \item \texttt{tmax = 5} ;
  \item recherche par candidats de \texttt{K} ;
  \item \texttt{dedupe = false}.
\end{itemize}

Cette pile est une approximation raisonnable du protocole du papier, mais pas
une reproduction exacte. Les principaux ecarts sont :
\begin{itemize}[leftmargin=*]
  \item absence du \texttt{Llama-7B} fine-tune des auteurs ;
  \item absence du sampling \texttt{convex} exact ;
  \item merge final encore simplifie ;
  \item prompting local necessairement different du protocole original.
\end{itemize}

\section{Essai d'amelioration : approximation du sampling \texttt{convex}}

Comme l'article souligne l'importance du \texttt{convex sampling}, une
approximation repo-locale de cette idee a ete implementee. Cette version
selectionne d'abord des points extremes via une projection basse dimension et
une coque convexe, puis complete l'echantillon avec une logique de
diversification.

Nous avons teste cette variante sur les benchmarks les plus en retard, comme le
montre le tableau~\ref{tab:convex-ablation}.

\begin{table}[h]
  \centering
  \begin{tabular}{lrrr}
    \toprule
    Benchmark & NMI random (\%) & NMI convex-proxy (\%) & Delta \\
    \midrule
    CLINC(I) & 90.90 & 90.11 & -0.79 \\
    Massive(I) & 74.02 & 72.53 & -1.49 \\
    \bottomrule
  \end{tabular}
  \caption{Ablation de l'approximation du sampling \texttt{convex}.}
  \label{tab:convex-ablation}
\end{table}

Cet essai montre que l'approximation actuelle du \texttt{convex sampling}
n'apporte pas d'amelioration mesurable sur ces benchmarks, et degrade meme
legerement les scores. Cela suggere que :
\begin{itemize}[leftmargin=*]
  \item soit notre approximation est encore trop eloignee du protocole
  original ;
  \item soit le sampling n'est pas le facteur principal limitant la
  reproduction dans notre implementation.
\end{itemize}

\section{Analyse des ecarts restants}

Les ecarts entre nos resultats et ceux du papier peuvent raisonnablement
s'expliquer par plusieurs facteurs cumulatifs.

\subsection{Difference de modele LLM}

Le papier utilise un \texttt{Llama-7B} fine-tune pour les benchmarks anglais,
alors que notre meilleure configuration executable repose sur
\texttt{Mistral-7B-Instruct}. Ce changement affecte probablement :
\begin{itemize}[leftmargin=*]
  \item la qualite des jugements \texttt{Good/Bad} ;
  \item la stabilite du namer ;
  \item la sensibilite aux formulations de prompts.
\end{itemize}

\subsection{Sampling encore approximatif}

Le papier signale explicitement l'interet du \texttt{convex sampling}. Notre
pipeline en propose maintenant une approximation, mais pas l'algorithme
original exact.

\subsection{Merge final simplifie}

Le merge utilise encore une logique plus simple que le merge probabiliste
discute dans le papier. Cela peut contribuer a la sur-fragmentation observee
sur certains benchmarks.

\subsection{Recherche de \texttt{K}}

Nos listes de \texttt{K} candidates restent heuristiques. Or le papier insiste
sur la selection dynamique de l'espace de recherche et sur l'importance de ce
point pour l'efficacite globale du pipeline.

\section{Bilan}

La reproduction obtenue est suffisamment solide pour soutenir les conclusions
suivantes :
\begin{itemize}[leftmargin=*]
  \item le coeur de l'approche du papier a bien ete reconstruit et execute de
  bout en bout ;
  \item la comparaison sur les benchmarks anglais est pertinente et
  techniquement defensible ;
  \item la methode reproduite reste proche des resultats du papier sur
  plusieurs jeux de donnees ;
  \item les ecarts restants sont localises, explicables, et peuvent faire
  l'objet d'une amelioration ciblee.
\end{itemize}

En particulier, la reproduction actuelle est deja forte sur Bank77 et MTOP(I),
et raisonnablement proche sur CLINC(I) et Massive(I) compte tenu des
contraintes d'acces aux modeles et des simplifications d'implementation.

\section{Suite du travail}

Au vu des resultats obtenus, la suite la plus logique du projet n'est plus de
relancer des benchmarks bruts, mais de cibler les leviers susceptibles de
reduire les ecarts restants :
\begin{enumerate}[leftmargin=*]
  \item ameliorer encore le comportement du couple coherence / naming ;
  \item reduire la sur-fragmentation des clusters finaux ;
  \item revisiter la recherche de \texttt{K} ;
  \item tester, si possible, une alternative plus proche du \texttt{Llama-7B}
  du papier.
\end{enumerate}

Ainsi, cette phase de reproduction constitue a la fois :
\begin{itemize}[leftmargin=*]
  \item une validation experimentale de la methode ;
  \item une base propre pour les ameliorations algorithmiques proposees dans la
  suite du projet.
\end{itemize}
